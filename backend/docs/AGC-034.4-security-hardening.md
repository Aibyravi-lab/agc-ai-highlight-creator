# AGC-034.4 — Security Hardening (MVP)

## Overview

Five security improvements added to the AGC backend upload pipeline and HTTP layer. No AI pipeline, authentication flow, or frontend changes.

---

## New Configuration Values

Added to `backend/app/config/config.py` (all overridable via `.env`):

| Variable | Default | Purpose |
|---|---|---|
| `DUPLICATE_UPLOAD_WINDOW_MINUTES` | `10` | Time window (minutes) for duplicate upload detection per user |

---

## Protections Implemented

### 1. MIME Type Validation

**Service:** `app/services/mime_validation_service.py`  
**Router:** `POST /upload/`

The first 12 bytes of every uploaded file are read and inspected against known video format magic byte signatures. The client-supplied `Content-Type` header is not trusted.

**Allowed MIME types:**

| MIME Type | Format | Detection |
|---|---|---|
| `video/mp4` | MP4 / M4V | `ftyp` box at offset 4 (non-QuickTime brand) |
| `video/quicktime` | MOV | `ftyp qt  ` box, or legacy `moov`/`mdat` atom |
| `video/x-msvideo` | AVI | `RIFF` magic + `AVI ` at offset 8 |
| `video/webm` | WebM | EBML magic `\x1a\x45\xdf\xa3` |
| `video/x-matroska` | MKV | EBML magic `\x1a\x45\xdf\xa3` |

Unsupported types are rejected with **HTTP 415** and error code `INVALID_MIME_TYPE`.

**No new pip dependencies** — detection is pure Python from the file header bytes.

---

### 2. Filename Sanitization

**Router:** `POST /upload/`

Filenames are sanitized before any file is written to disk.

**Rules applied (in order):**
1. `Path(filename).name` — strips any directory path component (prevents traversal like `../../etc/passwd`)
2. Unicode normalization (NFKD) then ASCII encoding — removes non-ASCII characters
3. Regex replacement of unsafe filesystem characters: `< > : " / \ | ? * \x00–\x1f`
4. Spaces replaced with underscores
5. Leading/trailing dots and underscores stripped
6. Windows reserved names (`CON`, `NUL`, `COM1`–`COM9`, `LPT1`–`LPT9`, etc.) prefixed with `upload_`
7. Empty result replaced with `upload`

The final filename stored on disk: `{8-char UUID hex}_{sanitized_stem}{extension}`

Filenames with no valid stem return **HTTP 400** with error code `INVALID_FILENAME`.

---

### 3. Duplicate Upload Protection

**Service:** `app/services/upload_cache_service.py`  
**Router:** `POST /upload/`

Duplicate uploads from the same authenticated user are detected within the configured time window.

**Detection key:** `(user_id, original_filename_lowercased, file_size_bytes)`

If an identical upload is detected within `DUPLICATE_UPLOAD_WINDOW_MINUTES`:
- **No file is written** to disk
- The original upload response is returned with `"duplicate": true` and code `DUPLICATE_UPLOAD`
- HTTP status remains **200** (not an error — the upload "succeeded" via the cached result)

**Implementation:** thread-safe in-memory dict with a `threading.Lock`. Entries expire automatically on next lookup. Cache is process-local (resets on server restart).

---

### 4. Standard Error Codes

All upload errors now return structured `detail` objects:

```json
{
  "detail": {
    "code": "ERROR_CODE",
    "message": "Human-readable description."
  }
}
```

**Error codes:**

| Code | Status | Trigger |
|---|---|---|
| `INVALID_FILENAME` | 400 | Missing or empty filename after path stripping |
| `INVALID_EXTENSION` | 400 | File extension not in whitelist |
| `UPLOAD_TOO_LARGE` | 413 | File exceeds `MAX_UPLOAD_SIZE_MB` |
| `INVALID_MIME_TYPE` | 415 | Magic bytes do not match any allowed video format |
| `DUPLICATE_UPLOAD` | 200 | Same user re-uploads identical file within window |
| `MAX_CONCURRENT_JOBS` | 429 | User has reached `MAX_CONCURRENT_JOBS_PER_USER` active jobs |

---

### 5. Security Headers

**File:** `app/main.py`

`Permissions-Policy` header added to the existing `security_headers_middleware`.

**Complete header set now applied to every response:**

| Header | Value | Purpose |
|---|---|---|
| `X-Content-Type-Options` | `nosniff` | Prevents MIME-type sniffing |
| `X-Frame-Options` | `SAMEORIGIN` | Prevents clickjacking |
| `Referrer-Policy` | `strict-origin-when-cross-origin` | Controls referrer leakage |
| `Permissions-Policy` | `camera=(), microphone=(), geolocation=()` | Disables sensitive browser APIs |
| `Strict-Transport-Security` | `max-age=31536000; includeSubDomains` | HTTPS enforcement (only when `HTTPS_ENABLED=true`) |

## Filename Sanitization Rules — Quick Reference

| Input | Output |
|---|---|
| `../../etc/passwd.mp4` | `passwd.mp4` (path stripped) → `8hex_passwd.mp4` |
| `my video (1).mp4` | `8hex_my_video__1_.mp4` |
| `café_stream.mp4` | `8hex_caf_stream.mp4` (non-ASCII dropped) |
| `CON.mp4` | `8hex_upload_CON.mp4` (reserved name prefixed) |
| `....mp4` | `8hex_upload.mp4` (empty stem fallback) |
| `hello world.mp4` | `8hex_hello_world.mp4` |

## Manual Testing Checklist

### 1. MIME Type Validation

- [ ] Upload a valid `.mp4` file — should succeed with HTTP 200 and `mime_type: "video/mp4"` in response
- [ ] Upload a valid `.mov` file — should succeed with `mime_type: "video/quicktime"`
- [ ] Upload a valid `.avi` file — should succeed with `mime_type: "video/x-msvideo"`
- [ ] Upload a valid `.webm` file — should succeed with `mime_type: "video/webm"`
- [ ] Upload a valid `.mkv` file — should succeed with `mime_type: "video/webm"` (EBML shared)
- [ ] Rename a `.jpg` file to `.mp4` and upload it — should receive **HTTP 415** with `code: "INVALID_MIME_TYPE"`
- [ ] Rename a `.txt` file to `.mp4` and upload it — should receive **HTTP 415**
- [ ] Upload a real `.mp4` saved with a `.mov` extension — should succeed (MIME detected from bytes, not extension)

### 2. Filename Sanitization

- [ ] Upload a file with spaces in the name (e.g. `my gameplay.mp4`) — filename stored should have underscores
- [ ] Upload a file with path separators (e.g. `../../evil.mp4`) — stored filename must not contain `..` or `/`
- [ ] Upload a file named `CON.mp4` (Windows reserved) — stored filename should be prefixed with `upload_`
- [ ] Upload a file with non-ASCII characters (e.g. `café.mp4`) — non-ASCII chars should be stripped or transliterated
- [ ] Upload a file named `.mp4` (empty stem) — should succeed with `upload.mp4` stem
- [ ] Verify the stored file path never contains `..` or absolute path segments from the original filename

### 3. Duplicate Upload Protection

- [ ] Upload the same file twice within 10 minutes — second response should include `"duplicate": true` and `"code": "DUPLICATE_UPLOAD"` but HTTP 200
- [ ] Verify only one physical file exists on disk after two identical uploads
- [ ] Upload the same file after 10 minutes (or set `DUPLICATE_UPLOAD_WINDOW_MINUTES=0` in `.env` and restart) — should create a new file
- [ ] Upload the same filename but different size — should NOT be detected as duplicate
- [ ] Upload the same filename+size from a different user account — should NOT be detected as duplicate (per-user isolation)

### 4. Error Code Structure

- [ ] Trigger each error and confirm response body has `{"detail": {"code": "...", "message": "..."}}`
- [ ] `INVALID_FILENAME` — POST with no file / empty filename
- [ ] `INVALID_EXTENSION` — Upload a `.exe` file
- [ ] `UPLOAD_TOO_LARGE` — Upload file exceeding `MAX_UPLOAD_SIZE_MB`
- [ ] `INVALID_MIME_TYPE` — Upload renamed non-video file
- [ ] `MAX_CONCURRENT_JOBS` — Start 3rd job when 2 are running

### 5. Security Headers

- [ ] Make any API request and inspect response headers
- [ ] Confirm `Permissions-Policy: camera=(), microphone=(), geolocation=()` is present
- [ ] Confirm existing headers are still present: `X-Content-Type-Options`, `X-Frame-Options`, `Referrer-Policy`
- [ ] With `HTTPS_ENABLED=false` (default), confirm `Strict-Transport-Security` is absent
- [ ] With `HTTPS_ENABLED=true`, confirm `Strict-Transport-Security` is present

### 6. Regression

- [ ] Normal upload flow still works end-to-end (upload → pipeline start → job completes)
- [ ] Upload response still contains `filename`, `original_filename`, `size_mb`, `location`
- [ ] Pipeline 429 response now includes `{"code": "MAX_CONCURRENT_JOBS", "message": "..."}` (confirm frontend handles dict detail if applicable)
- [ ] `.webm` files can now be uploaded (new extension in whitelist)
- [ ] `.mp4`, `.mov`, `.mkv`, `.avi` files still accepted
